## 0. 执行摘要

### 0.1 整体"拿现成 vs 自研"占比判断(已按评审下修)

按"代码量"与"风险面"两个维度评估,arclightagent 的全栈构成大致是:

- **约 80% 拿现成**(原清单写 85%,评审指出前端产品工程与主循环被系统性折叠进"薄接缝",真实自研占比更高,下修至约 80%):provider 调用、流式、MCP、schema、编辑器、文档生成、浏览器自动化、云浏览器、搜索抓取、cron、推送、向量/全文检索、ORM、PTY、git、tree-sitter、LSP——这些**核心能力件全部由成熟 npm 库/SDK/二进制承载**,几乎零自研。
- **约 20% 自研薄接缝 + 产品级工程**(原写 15%):agent 主循环、工具元数据调度、MCP 安全审计 + 凭证代理、Skill/Hook 加载器、压缩、记忆混合检索、CitationAgent、ResearchOrchestrator、RepoMap、Plan/Act、心跳协调、各端 transport 桥接、截图通道、权限审批,**以及评审点名的两处真·产品级工程:agent 主循环(心脏)与聊天前端(assistant-ui 工具渲染 + 权限 UI)**。这部分仍无一是"造轮子",但其中两处不是"薄接缝"而是核心工程,见 §5。

**一句话**:本项目几乎没有"可整体安装的 agent 框架"能用(也刻意不用 LangGraph/Mastra),但**每一个底层能力件都有成熟现成实现**;自研发生在"内核 async-generator 事件模型 ↔ 现成库"的接缝处——其中**主循环与前端两处是真工程量,不可当薄接缝低估**(评审修订)。

### 0.2 最该警惕的"伪轻量"(看似拿来即用,实则要大量缝合)

按踩坑严重度排序。**评审新增 #11–#14,并对 #1/#3 的诚实标注予以保留确认**:

| # | "伪轻量"陷阱 | 真相 | 正确姿势 / 回退 |
|---|---|---|---|
| 1 | **Vercel AI SDK 的 "agent 循环"** | `streamText({ stopWhen })` 只是单 turn 工具循环原语,**不提供可中断/压缩边界/steering 队列/per-turn 重建**。被 marketing 包装成"agent 框架"。 | 当作 turn 原语,顶层 async-generator 主循环自研(评审修订:**实评 800-1500 行,非 300-400**);**回退**:所有对 `ai` 的依赖收敛进单一 adapter 层 |
| 2 | **resumable-stream(断线恢复)** | **本子系统 bug 密度最高处**:引 Redis 运维面、active-stream 生命周期/清理/幽灵流、replay 去重 + epoch 合并 UX 复杂。 | MVP **不引**;只做"刷新不丢"最朴素版(SQLite event 表 + 重连续推),Redis 版后置阶段二 |
| 3 | **open_deep_research / GPT-Researcher / browser-use** | 看似"现成 deep research / 浏览器自动化",**全是 Python**,引入 = Python sidecar + 跨语言边界。 | 仅借设计,TS 侧用 Vercel AI SDK 复现 pipeline。**评审修订:opensquilla 自带 `deep-research` SKILL.md(MIT-0)可直接复用,见 §1.4——不必从零复现** |

| 9 | **LiteLLM Proxy 版本迭代** | 一年 1349+ 版本,配置格式偶变,路由失败是静默 fallback。 | 锁定小版本;MVP 单 provider 时可不起 Proxy,AI SDK 直连 Anthropic |
| 10 | **MV3 Service Worker 持久连接** | SW 30s 休眠,不可维持传统持久连接。 | `chrome.alarms` 唤醒 + WS keepalive + 状态持久化恢复 |
| 11 | **`@assistant-ui/react` 标"拿来即用"**(评审新增) | 把内核 ArcEvent 桥进 `ExternalStore`/`AISDKRuntime` + Monaco diff 渲染 + 终端流渲染 + 流式 part 级更新 + 权限审批模态双向往返 = **产品级前端工程,非 200 行接缝**。 | 认领真实工程量(实评 1500-3000+ 行,见 §5);**回退**:工具渲染先用纯文本/JSON 降级,Monaco/终端富渲染分期上 |
| 12 | **Bun + 原生模块**(评审新增,runtime 级暗雷) | `node-pty` / `sqlite-vec`(原生扩展)/ `@napi-rs/keyring` 在 Bun N-API 下兼容性历来不稳;任一挂掉=降级 Node 或自编。清单 Bun-first 却逐个轻描淡写。 | **MVP 第一周强制跑通"Bun + node-pty + sqlite-vec + @napi-rs/keyring" smoke test**;**回退**:任一不通则该 runtime 段降级 Node 子进程或换纯 JS 实现 |
| 13 | **依赖宿主二进制的"现成件"**(评审新增) | `simple-git`/`node-pty`/LSP/xelatex 依赖宿主有 `git`/`tsserver`/`pyright`/`xelatex`;单机 MVP 没问题,**上远程沙箱=镜像要预装这些**,是镜像维护/体积成本,非 npm install 可解。 | 单机 MVP 接受;沙箱阶段把二进制预装列入镜像清单,不计入"拿来即用" |
| 14 | **社区 MCP server(Google Calendar/Gmail)**(评审新增,估值不一致) | 是 #3 Tool Poisoning(CVE-2025-54136)主要入口;审查第三方 server 工具描述 + 鉴权隔离 + 凭证不下发**远不止"轻度封装"**,与内核子系统把同一风险列为安全关键自研接缝(~150 行)**估值不一致**。 | 规划子系统统一按"安全关键自研"对待,复用内核 MCP 审计 + 凭证代理,不另算"轻度封装" |
