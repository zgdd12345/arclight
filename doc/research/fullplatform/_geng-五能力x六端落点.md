# arclightagent 全平台架构详设 ——【五大能力 × 六端 落点矩阵】

> 本文是 ARCHITECTURE_BLUEPRINT.md 的下沉详设,锚定其§6(五大能力模块设计)/§7(跨平台策略)/§8(决策表)。**严格继承主蓝图全部已拍板决策,不另起炉灶**:同一 headless 内核服务(Bun+Hono,TS/Bun)+ 各端薄客户端;网页优先;桌面/移动用 Tauri 2.0(非 Electron);沙箱默认本地(nono/系统 bwrap exec)、SaaS opt-in、浏览器用 Browserbase/Steel 不叠 E2B;凭证一律沙箱外签名放行;截图走独立二进制 WS/WebRTC;不 vendoring LGPL bwrap。
>
> **六端定义**:Web(Next.js,先发)/ 桌面(Tauri2)/ 移动(Tauri2 iOS+Android)/ CLI(Bun --compile + OpenTUI)/ VSCode 插件(Chat Participants + MCP)/ Chrome 扩展(MV3,side panel + service worker)。
>
> **档位定义**:
> - **主场**=该端是此能力的最优/天然载体,UX 上限最高,优先在此打磨。
> - **可用**=能力完整或接近完整可用,体验良好,非最优载体。
> - **裁剪**=受端能力/平台限制做功能删减或改走远程/opt-in,核心可用但有阉割。
> - **不适合**=平台原理性不匹配,不投入或仅做最小入口/转发,强行做是净负。

## 0. 贯穿全文的总纪律(六端共性,先讲清)

1. **内核零 UI 依赖、各端只换壳**(§7 核心原则)。五大能力的 Agent 主循环、工具系统、上下文压缩、记忆、权限、持久化、沙箱抽象、provider 网关**全部在内核**,六端**不重写任何能力逻辑**,只负责:输入采集 + 事件流渲染 + 端特有执行后端的接驳。
2. **协议**:MVP 单 repo 共享 TS 类型 + HTTP/SSE;第二端(即本详设涉及的 CLI/桌面/插件/扩展,§9 阶段五)起上 OpenAPI→TS SDK 自动生成 + AG-UI 适配器(预留自建流式 codegen 预算)。**Web 之外的端都是"第二端"基建的消费者。**
3. **沙箱与凭证是端无关的硬边界**:无论哪个端发起,代码执行默认走内核侧本地沙箱;**凭证永远在沙箱外、按动作签名放行,沙箱内零真实凭证**。端只是观测/控制面板,不持有真实凭证执行域。
4. **移动端原理性约束(贯穿五能力)**:iOS/Android **无本地代码沙箱、无本地浏览器自动化驱动(无 CDP/Playwright)、后台执行受系统严格限制**。因此移动端凡涉及"执行"的能力,一律**走远程内核 server + opt-in 远程沙箱/云浏览器**,移动壳只做输入 + 观测 + 审批 + 系统通知。
5. **Chrome MV3 原理性约束**:**禁 eval / 禁远程 JS**,service worker 非常驻(易被回收)。故扩展内**不跑推理、不跑沙箱**,一律 WS 连本地内核中转;但**扩展运行在页面上下文内,是 computer-use 的天然主场**。
6. **模型分层纪律**(§9):各端发起的同一能力共享内核的模型路由策略,端不自选模型层级。

## 1. 写代码

> 主蓝图定位:★MVP 唯一能力(§6.1)。RepoMap(tree-sitter+PageRank)+ SEARCH/REPLACE + LSP + 反射验证闭环 + 本地沙箱默认 + shadow-git 检查点。这是与"软件工程载体"(IDE/终端)强耦合的能力。

### 1.1 各端档位与理由

| 端 | 档位 | 理由 |
|---|---|---|
| **Web** | **主场(MVP 先发)** | 蓝图指定 MVP 唯一交付端。Monaco/CodeMirror diff + 命名端口 iframe 内嵌沙箱终端 + SSE 16ms coalescing,完整呈现 RepoMap/编辑/反射闭环。 |
| **VSCode** | **主场(与 Web 并列,P3)** | 写代码的"原生工作台"。能直接复用编辑器的 **diff 视图 / LSP / 文件树 / SCM**,无需自建 IDE 表层。走 Chat Participants(@arclightagent)+ MCP server 注册,内核编辑能力以 diff 形式落到 VSCode WorkspaceEdit。**LSP/诊断走 VSCode 原生**而非内核内置 LSP,是此端最大适配差异。 |
| **桌面** | **可用(P3,先 PWA 过渡)** | Tauri WebView 复用 Web 前端全部代码,额外得到本地文件系统直读 + 本地内核 sidecar(spawn)→ 沙箱可走本地真实文件而非云端 iframe。体验接近 Web 主场,但缺 VSCode 的成熟编辑器生态。 |
| **CLI** | **可用(P2)** | 终端是开发者第二自然栖息地。OpenTUI 渲染编辑计划/diff/test 输出;反射验证闭环在终端可读性好。**裁剪点**:diff 用 TUI 文本 diff 而非图形 diff;无内嵌 IDE,沙箱终端即本地终端。 |
| **移动** | **裁剪(远程 opt-in)** | **移动无本地沙箱**→代码执行只能走远程 internal server + opt-in 远程沙箱(E2B/Vercel)。移动端定位为**审阅/批准/轻量改动 + 触发远程任务**:看 diff、批 PR、跑远程测试、收结果通知。**不做本地编辑/本地执行**。 |
| **Chrome** | **不适合** | 浏览器扩展无文件系统/编译/LSP/沙箱栖息地,与代码工作流原理性不匹配。仅保留**最小入口**:在 GitHub/GitLab 页面唤起内核对当前 repo/PR 做只读分析(走内核+远程),不做编辑执行。 |

### 1.2 各端能力裁剪与适配要点

- **Web(主场)**:Monaco diff;沙箱命名端口 iframe 内嵌终端(OpenHands AGENT_SERVER/VSCODE/WORKER 约定);SSE 增量 + 16ms coalescing;shadow-git 检查点 /undo 做成可视化时间轴。
- **VSCode(主场,端特定后端)**:**编辑走 VSCode WorkspaceEdit + 原生 diff/SCM,LSP 走 VSCode language server**(内核内置 LSP 在此端让位),诊断/findReferences 复用编辑器。审批用 VSCode 原生 QuickPick/Modal。这是"端特定执行后端"的典型——内核出编辑意图,VSCode 落地。
- **桌面**:本地内核 sidecar + 本地文件系统;沙箱默认本地 nono/系统 bwrap exec(与内核同机,无 iframe 跨端开销)。
- **CLI**:OpenTUI 文本 diff;反射闭环(edit→lint/test→自校正)终端流式;shadow-git /undo 为命令。
- **移动**:**纯远程**——无本地沙箱(蓝图明文:移动端无本地沙箱→代码执行只能走远程 opt-in)。展示 diff、审批高危编辑、触发/查看远程测试结果、推送通知。
- **Chrome**:仅代码托管页面的只读分析入口,转发内核。

### 1.3 内核共享 vs 端特定后端

- **跨端共享内核逻辑**(仅 UI/输入差异):RepoMap、SEARCH/REPLACE 多格式编辑引擎、反射验证闭环、shadow-git 检查点、Git 集成、沙箱抽象——**一次建,Web/桌面/CLI/移动/Chrome 全消费同一份**。
- **需端特定执行后端**:① **VSCode**——diff/LSP/SCM 走编辑器原生 API(端特定落地后端);② **移动**——执行后端强制为远程 opt-in 沙箱(无本地后端);③ 桌面/CLI 共用内核本地沙箱后端。

## 2. 写文章

> 主蓝图定位:阶段二自建流水线(§6.2)。opensquilla paper-*(Apache-2.0)移植为 SKILL.md + MetaGPT ActionNode 结构化输出 + 富文本编辑器 + 分阶段审批 UI + 文档生成(docx/pptx/pdf/latex)。蓝图明示:**网页端是写作的最佳载体,远胜 CLI**。

### 2.1 各端档位与理由

| 端 | 档位 | 理由 |
|---|---|---|
| **Web** | **主场** | 蓝图明文"写作能力的最佳载体"。所见即所得富文本/Markdown 编辑器、章节级流式渲染(marked+shiki)、大纲→草稿→精修分阶段可审批 UI、引用可点击溯源脚注、长文 background 异步 + SSE 增量推章节。 |
| **桌面** | **主场(与 Web 并列)** | Tauri WebView 复用 Web 富文本编辑器全部代码,额外得**本地文档库直读 + 离线草稿 + 系统级文件保存对话框**,长文写作 + 本地资料管理优于纯 Web。 |
| **移动** | **裁剪** | 触屏 + 小屏不适合长文重度编辑/多栏审批,但适合**口述成稿(语音输入)+ 阅读审阅 + 单段精修 + 批准章节**。写作流水线全在远程内核跑,移动端订阅 SSE 看章节流出、做轻量审批。**裁剪掉**:复杂多栏审批 UI、大纲拖拽重排。 |
| **CLI** | **裁剪** | 蓝图明示 CLI 远逊于 Web。终端不适合富文本/所见即所得,但适合**触发写作流水线 + 流式看大纲/章节文本 + 输出文件路径**。定位为"批处理式写作":一句话需求→后台生成→落 .md/.docx。**裁剪掉**:富文本编辑、可视化分阶段审批(降级为终端 y/n 审批)。 |
| **VSCode** | **可用** | 技术文档/README/博客等 **Markdown-as-code** 写作场景下,VSCode 是合理载体:用编辑器原生 Markdown 预览 + diff 看精修改动 + SCM 版本化。但通用富文本(docx/pptx 排版)非其所长。 |
| **Chrome** | **裁剪** | 天然场景:**网页内取材 + 划词成稿 + 在线编辑器(Docs/Notion/邮件)旁辅助写作**。side panel 做大纲/草稿助手,把选中网页内容喂给内核写作流水线。**裁剪掉**:完整文档生成(docx/pptx 后端在内核,扩展只触发+下载)。 |

### 2.2 各端能力裁剪与适配要点

- **Web/桌面(主场)**:完整 paper-* 流水线 UI——大纲→草稿→精修→引用四阶段可视化审批;章节级流式;可点击溯源脚注组件;文档生成后端(内核侧 docx/pptx/pdf/latex)出文件供下载/本地保存。桌面额外:本地文档库、离线、系统保存对话框。
- **移动**:语音输入成稿 + 阅读审阅 + 单段精修 + 章节级 SSE 订阅 + Web Push 通知"长文已生成"。
- **CLI**:OpenTUI 流式大纲/章节文本;终端审批降级为 y/n;输出文件路径。
- **VSCode**:Markdown 写作走编辑器原生预览 + diff + SCM。
- **Chrome**:side panel 划词取材 → 内核写作流水线;文档生成在内核,扩展触发并下载。

### 2.3 内核共享 vs 端特定后端

- **跨端共享内核逻辑**(纯 UI/输入差异):paper-* 写作流水线 SKILL、ActionNode 结构化输出引擎、自校正闭环、结构化压缩模板、文档生成后端、引用溯源数据——**全部内核共享**,六端差异**仅在编辑器/审批 UI 与输入方式**(键盘/语音/划词)。
- **端特定后端**:无强制端特定执行后端(写作不依赖端侧沙箱/系统能力)。文档生成统一在内核;唯一端差异是**呈现层**(富文本 vs Markdown-as-code vs TUI vs side panel)。**这是五能力中"跨端共享内核、仅 UI 差异"最纯粹的一项。**

## 3. 调研(Deep Research)

> 主蓝图定位:阶段三独立 MVP(§6.3)。Orchestrator-Subagent(LeadResearcher Opus + 并行子代理 Sonnet/Haiku 独立上下文)+ web_search/web_fetch + MCP 数据源 + 独立 CitationAgent(句级溯源)+ 异步任务持久化 + 断点续研。本质是**长时(3-30 分钟)异步后台任务**,对端的核心要求是"流式反馈 + 断点续研 + 通知"。

### 3.1 各端档位与理由

| 端 | 档位 | 理由 |
|---|---|---|
| **Web** | **主场** | 调研面板:background 异步 + SSE 推 thought/搜索进度/子报告;规划阶段流式展示 subtopics 供审批(fan-out 前 HITL);任务持久化 + 断点续研;报告引用可点击溯源。大屏适合读长报告 + 审阅引用。 |
| **桌面** | **主场(与 Web 并列)** | 复用 Web 调研面板 + 系统通知(任务完成不必盯着浏览器)+ 本地报告归档。长时任务 + 系统级通知是桌面优势。 |
| **移动** | **可用** | 调研是**异步后台任务**,与移动端"碎片化 + 通知驱动"天然契合:发起调研→锁屏→**系统通知/Web Push 告知完成**→回来读报告 + 审批计划。**裁剪点**:超长报告小屏阅读体验受限、引用溯源交互简化;但发起/审批/读结论闭环完整,故评"可用"而非裁剪。 |
| **CLI** | **可用** | 终端发起调研 + 流式看 subtopics/检索进度 + 落 .md 报告。开发者/脚本化调研友好。**裁剪点**:可点击引用溯源降级为脚注引用列表 + 链接。 |
| **VSCode** | **裁剪** | 可在 Chat Participant 里发起调研 + 报告以 Markdown 落工作区,适合"为当前项目做技术调研"。但非调研最佳载体(无专门审阅 UI)。**裁剪掉**:流式 subtopics 可视化审批(降级为聊天内确认)。 |
| **Chrome** | **裁剪(天然取材入口)** | side panel 对**当前页面/选中内容发起溯源式调研**是天然场景(就地深挖)。但长报告呈现/断点续研在扩展内不理想(service worker 非常驻)→ 发起后转 Web/桌面看结果。**裁剪掉**:长任务驻留(交给内核 + 通知)。 |

### 3.2 各端能力裁剪与适配要点

- **Web/桌面(主场)**:规划阶段流式 subtopics 审批面板;thought/进度/子报告 SSE;可点击引用溯源;断点续研 + 前端重连状态恢复;桌面加系统通知 + 本地归档。
- **移动**:发起 + 计划审批(轻量)+ Web Push 完成通知 + 读报告。任务全在远程内核,移动断网不影响后台调研。
- **CLI**:OpenTUI 流式进度 + 落 .md;引用为脚注列表。
- **VSCode**:Chat 内发起 + 报告落工作区 Markdown。
- **Chrome**:side panel 就地对页面取材发起,转 Web/桌面看长报告。
- **共性**:**所有端必须做任务持久化 + 断点续研**(蓝图硬要求),否则网络波动丢结果——这正是把任务放内核、端只订阅的根本理由。

### 3.3 内核共享 vs 端特定后端

- **跨端共享内核逻辑**(纯 UI/输入差异):Orchestrator-Subagent、并行检索子代理、CitationAgent、web_search/web_fetch、MCP 数据源、异步任务持久化、断点续研——**完全内核共享**。端差异仅在**进度呈现 + 计划审批 UI + 完成通知方式**。
- **端特定后端**:无端特定执行后端。唯一端能力差异是**通知通道**(Web Push / 系统通知 / 终端打印 / 扩展通知),通知是"端能力"而非"执行后端"。调研与写作同属"跨端共享内核、仅 UI/通知差异"类。

## 4. Computer Use

> 主蓝图定位:阶段四自建(§6.4),11 仓覆盖最弱。DOM/AX 优先 + 视觉兜底;Stagehand v3 执行层;Browserbase/Steel(已隔离,不叠 E2B);截图走独立二进制 WS/WebRTC(JPEG/WebP 帧差)或云浏览器 CDP/VNC;**硬边界=沙箱内零凭证 + 凭证代理外置签名放行 + HITL 强确认 + 域名白名单 + 审计**。蓝图明示全平台扩展路径:web 循环打牢后复用到桌面(OSWorld)/Android(AndroidWorld),**差异仅执行后端**。

### 4.1 各端档位与理由

| 端 | 档位 | 理由 |
|---|---|---|
| **Chrome** | **主场(天然主场)** | 扩展运行在页面上下文内,**对当前真实浏览器会话的 DOM/AX 树天然可达**,无需云浏览器即可就地操控用户已登录的页面。这是 computer-use 最自然的载体——用户看着 agent 在自己浏览器里操作。MV3 限制(无 eval/SW 非常驻)由"推理+决策在内核、扩展只做 DOM 动作执行+截图采集"绕过。 |
| **Web** | **主场(蓝图先验证端)** | 蓝图把 web computer-use 作为打牢"截图→推理→动作"闭环的首验证端。浏览器跑在**云沙箱(Browserbase/Steel)**,网页只是观测+控制面板;截图走独立二进制 WS/WebRTC(JPEG/WebP 帧差),高保真长会话用 CDP/VNC 远程渲染。 |
| **桌面** | **可用→裁剪** | 桌面是**OS 级 GUI computer-use**(非仅浏览器)的潜在载体(蓝图 OSWorld 路径),但桌面坐标级控制蓝图定性为"需自建 + 必须跑沙箱内",阶段更靠后。当前评:**浏览器 computer-use 可用**(复用 Web 方案 + 系统通知),**OS 级 GUI 控制裁剪**(高风险,远程沙箱内 + 强 HITL,非默认)。 |
| **移动** | **裁剪(远程 opt-in)** | **移动无本地浏览器自动化驱动(无 CDP/Playwright)**→必须走远程云浏览器(Browserbase/Steel)。移动端是**观测 + 审批面板**:看远程浏览器截图流、批高危动作。蓝图 AndroidWorld(本机 OS 控制)是更后置的独立后端,当前不在范围。**裁剪掉**:本机执行。 |
| **CLI** | **不适合** | 终端无视觉呈现载体,看不了截图流/无法做可视化 HITL 确认。仅保留**触发 + 文本日志 + 文本式审批**最小能力,不作为 computer-use 端。 |
| **VSCode** | **不适合** | IDE 非浏览器/GUI 操控场景,与 computer-use 原理性不匹配。仅可在 Chat 里转发触发,不作为载体。 |

### 4.2 各端能力裁剪与适配要点

- **Chrome(天然主场,端特定后端)**:**扩展即执行后端**——content script 执行 DOM/AX 动作(click/type/scroll/extract),采集 AX 树快照 + 视觉兜底截图,经 WS 回传内核;**推理/决策/规划全在内核**(MV3 禁 eval,不在扩展跑模型)。凭证仍走沙箱外签名放行——即便操作用户已登录页面,高危动作(支付/删除/外发)仍强制 HITL + 域名白名单 + 审计。
- **Web(云浏览器后端)**:浏览器在 Browserbase/Steel;截图独立二进制 WS/WebRTC + JPEG/WebP 帧差;高危确认走前端模态对话框(cline bridgePermissionCallbacks 思路)。
- **桌面**:浏览器 computer-use 复用 Web 方案;OS 级 GUI 控制裁剪为远程沙箱内 + 强 HITL(后置)。
- **移动**:远程云浏览器截图流观测 + 审批,无本机执行。
- **共性硬边界(端无关)**:**沙箱内零真实凭证 + 凭证代理外置按动作签名放行**;prompt injection 不可彻底解决,进程沙箱只限爆炸半径,真正硬边界是凭证不在执行域;HITL/白名单/审计为纵深补充。**此边界对六端一视同仁**。

### 4.3 内核共享 vs 端特定后端

- **跨端共享内核逻辑**:感知(DOM/AX 优先 + OmniParser 视觉兜底)、推理/动作规划、模型层(Claude CU/Gemini CU)、凭证签名放行、HITL/白名单/审计策略——**全部内核共享**。蓝图明示"web 循环打牢后复用到桌面/Android,差异仅执行后端"。
- **需端特定执行后端(本能力是端特定后端差异最大的一项)**:① **Chrome 扩展**——content script DOM 动作执行(就地操控真实会话,天然后端);② **Web/桌面/移动**——远程云浏览器(Browserbase/Steel)CDP 后端;③ **桌面 OS 级**——OS 注入后端(后置);④ **Android**——ADB/OS 注入后端(蓝图 AndroidWorld,后置)。**执行后端按端可插拔,感知/推理/安全闭环不变**——这正是蓝图"差异仅执行后端"的落地。

## 5. 日常规划

> 主蓝图定位:阶段五自建(§6.5),生活域全仓空白。Plan/Act 双模式 + 结构化 TODO(FocusChain)+ Cron 调度器 + 主动提醒(心跳协调器:事件合并 + 优先级带 + 活跃时段掩码,防刷屏/夜间打扰)+ 生活域工具(MCP 接 Google Calendar/Gmail,凭证沙箱外)+ 长期记忆(此阶段才评估 mem0/Zep + Dream)。本质是**主动性 + 通知驱动 + 跨设备常驻**的能力,对端的核心要求是"系统通知 + 随身可达"。

### 5.1 各端档位与理由

| 端 | 档位 | 理由 |
|---|---|---|
| **移动** | **主场** | 日常规划是**随身、通知驱动、生活域**能力,移动是最贴合载体:**系统通知(iOS/Android 原生推送)**做主动提醒、随时随地查看日历视图/checklist、碎片时间批准 agent 行动。蓝图主动提醒经"系统通知/Web Push"——移动端原生通知是最佳触达。 |
| **桌面** | **主场(与移动并列)** | 桌面常开 + **系统级通知/托盘**,工作时段的日程提醒、每日简报弹窗、看板/checklist 大屏管理优于移动。Tauri 系统通知 + 后台常驻 + 心跳 poll 适配良好。蓝图明示"日常规划在桌面/移动靠系统通知/Web Push"。 |
| **Web** | **可用** | 日历视图 + 看板/checklist UI 在 Web 完整呈现;主动提醒经 **Web Push**;durable session + 心跳 poll 断连恢复。**裁剪点**:浏览器关闭后无后台常驻,通知依赖 Web Push 且不如原生稳——故评"可用"非主场。 |
| **VSCode** | **裁剪** | 仅适合**开发场景的软件任务规划**(Plan/Act 双模式 + FocusChain task_progress,这正是 11 仓原生强项),不适合生活域日历/提醒。**裁剪掉**:日历视图、生活域 MCP 工具、主动生活提醒。定位为"编码任务的计划-待办"。 |
| **CLI** | **裁剪** | 同 VSCode——擅长软件任务的 Plan/Act + checklist + cron 触发(终端原生),不适合生活域可视化与主动提醒(终端无常驻通知)。**裁剪掉**:日历视图、主动推送(降级为下次进终端时打印待办)。 |
| **Chrome** | **不适合** | 浏览器扩展无系统级常驻通知能力(SW 非常驻)、无随身性,与"主动提醒 + 生活域"原理性不匹配。仅可在 side panel 看只读今日待办,不作为规划载体。 |

### 5.2 各端能力裁剪与适配要点

- **移动(主场)**:**iOS/Android 原生系统通知**做主动提醒(心跳协调器的活跃时段掩码→夜间不打扰直接映射到系统勿扰);日历视图 + checklist 触屏管理;碎片时间审批 agent 行动;durable 输入支持手机/Web 并发派活。
- **桌面(主场)**:Tauri 系统通知 + 托盘 + 后台常驻心跳;每日简报弹窗;看板/checklist 大屏。
- **Web**:日历/看板/checklist 完整 UI;Web Push 提醒;心跳 poll 断连恢复。**通知稳定性弱于原生**是其降为"可用"的根因。
- **VSCode/CLI**:仅软件任务 Plan/Act + FocusChain + cron;无生活域、无主动生活提醒。
- **生活域工具(端无关)**:日历/邮件经 **MCP 接 Google Calendar/Gmail**,**凭证沙箱外、按动作签名放行**——六端共用同一内核 MCP 接入,端不持有 Google 凭证。

### 5.3 内核共享 vs 端特定后端

- **跨端共享内核逻辑**:Plan/Act 双模式、FocusChain 结构化 TODO、Cron 调度器、心跳协调器(事件合并/优先级带/活跃时段掩码)、生活域 MCP 工具、长期记忆(mem0/Zep + Dream)——**全部内核共享**。规划/调度/提醒决策在内核产生。
- **需端特定后端(通知触达是关键端差异)**:主动提醒的**通知投递通道**是端特定的——**移动:iOS/Android 原生推送(APNs/FCM)**;**桌面:Tauri 系统通知/托盘**;**Web:Web Push**;CLI:终端打印;VSCode:编辑器通知。**心跳协调器在内核统一产出"何时提醒、是否夜间静默",端只负责按本端通知通道投递**——这是"内核共享决策 + 端特定投递后端"的清晰分工。生活域执行(改日历/发邮件)统一走内核 MCP,无端特定执行后端。

## 7. 跨端共享内核 vs 端特定执行后端(总结论)

### 7.1 跨端共享内核逻辑、仅 UI/输入/通知差异(端是"纯薄壳")

这些能力的**全部业务逻辑在内核**,六端差异仅在呈现/输入/通知,**新增一个端不需要重写能力**:

- **写文章**:最纯粹——paper-* 流水线 + ActionNode + 文档生成全内核,端仅换编辑器/审批 UI 与输入方式(键盘/语音/划词)。无任何端特定执行后端。
- **调研**:Orchestrator-Subagent + CitationAgent + 持久化 + 断点续研全内核,端仅换进度呈现 + 计划审批 UI + 完成通知通道。无端特定执行后端。
- **(共享件)** Agent 主循环、工具系统、上下文/记忆压缩、provider 网关、会话持久化、权限策略、Skills/MCP/Hooks 扩展系统——**六端零重写,全部内核共享**(蓝图§6.6)。

### 7.2 需端特定执行后端(内核出"意图/决策",端落地"执行/投递")

这些能力**内核统一决策,但执行/落地依赖端能力**,新增端需实现该端的后端适配器:

| 能力 | 端特定后端 | 各端落地 |
|---|---|---|
| **写代码** | **编辑落地 + 沙箱执行后端** | VSCode:WorkspaceEdit + 原生 LSP/diff/SCM(端特定落地)；桌面/CLI:内核本地沙箱;**移动:远程 opt-in 沙箱(无本地后端)**；Web:云 iframe 沙箱 |
| **computer use** | **GUI 操控执行后端(差异最大)** | **Chrome:content script DOM 动作(天然后端,就地操控真实会话)**；Web/桌面/移动:远程云浏览器 CDP(Browserbase/Steel)；桌面 OS 级/Android：OS 注入/ADB(后置)。**感知/推理/安全闭环不变,仅后端可插拔**(蓝图"差异仅执行后端") |
| **日常规划** | **主动提醒通知投递后端** | **移动:APNs/FCM 原生推送**；桌面:Tauri 系统通知/托盘；Web:Web Push；CLI:终端打印；VSCode:编辑器通知。**心跳协调器在内核决定"何时/是否夜间静默",端仅按本端通道投递**。生活域执行(改日历/发邮件)统一走内核 MCP,无端差异 |

### 7.3 一句话收口

**写文章 / 调研 = "跨端共享内核、仅 UI 差异"的纯薄壳能力**;**写代码 / computer use / 日常规划 = "内核共享决策 + 端特定后端落地"能力**,其端特定后端分别是【编辑/沙箱执行后端】【GUI 操控执行后端】【通知投递后端】。三类端特定后端均**可插拔、不改内核感知/推理/安全闭环**,与主蓝图§7"同一内核服务、各端只换壳"和§6.4"差异仅执行后端"的决策完全一致;**移动端无本地沙箱/无本地浏览器驱动、Chrome MV3 无 eval/SW 非常驻**这两条原理性约束,是本矩阵所有"裁剪/远程 opt-in/天然主场"判定的根因。
