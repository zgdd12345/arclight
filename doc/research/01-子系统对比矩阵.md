## 1. Agent 主循环

| 仓 | 做法 | 关键点 |
|---|---|---|
| codex | 外层 task 循环 + 内层 turn 采样循环（run_turn） | turn 前 `run_pre_sampling_compact` 预压缩；一个 Session 只跑一个 Task |
| opencode | `runner/llm.ts`：每 turn 严格一次 `llm.stream`，工具 FiberSet 并发 settle | Effect + `uninterruptibleMask` 保证工具结算中断安全；MAX_STEPS=25 |
| claudecode | `query()→queryLoop()` 双层 **async generator**，`while(true)` | 流式 token/工具进度/压缩边界/子代理输出统一成一条可中断事件流 |
| gemini-cli | `Turn.run()` 返回 AsyncGenerator（16 种事件枚举）+ LocalAgentExecutor 多轮 | 标准 ReAct；事件驱动，多端同接口消费 |
| pi | `agent-loop.ts` **纯函数式**双层循环 + steering/follow-up 双队列 | 无状态可测；Agent 类做有状态包装 |
| opensquilla | `engine/agent.py` 显式状态机 + `_turn_generator` 逐事件 yield；TurnRunner 10-stage pipeline | history 清洗链 + KV-cache 前缀稳定化 |
| cline | `recursivelyMakeClineRequests()` while 循环 + Mutex 序列化状态 | 无工具调用则注入 noToolsUsed 强制继续 |
| qwen-code | AgentCore 无状态 runLoop + Headless/Interactive 组合 | CoreToolScheduler 并发调度 |
| MetaGPT | RoleZero `_observe→_think→_act` + quick_think 意图分流 | TeamLeader 中枢路由 |
| OpenHands | 编排器（app_server）拉沙箱 + 执行器（sdk）跑真正循环 | 主循环在外部 SDK，本仓只编排 |
| aider | `run_one()` 同步 while + 反射循环（max 3） | 无 async，阻塞主循环 |

**best_pick：claudecode 的 async-generator 主循环**（理由：把"流式输出 / 工具进度 / 压缩边界 / 子代理通知 / 中断"统一成一条可 `.return()` 提前结束、可 `abortController` 中断的事件流，是最干净、最易插入压缩/权限/中断逻辑的范式）。**结构上配 pi 的"纯函数循环 + 有状态包装"分层**（可测试性最佳）。网页端把这条事件流直接映射成 SSE/WebSocket 帧即可。

- 调研/大型任务需要 fan-out 并行，codex 的"主协调-子执行 + 角色模板 + thread_spawn_depth 防递归"是可直接复用的骨架。
- claudecode 的 **fork 共享父 cache** 是上下文经济学高阶玩法（既隔离子任务噪声又不重建昂贵 cache），且 coordinator 把"综合理解必须自己做、验证必须独立 PASS/FAIL"写进编排协议，直接对治多代理"互相鼓掌、幻觉完成"通病。
- **消息总线/路由范式参考 MetaGPT 的 TeamLeader + send_to/cause_by 双过滤**（单一 orchestrator + 去中心化路由，适合"主控 + 子能力"编排）。
- 跨网络分布式 agent 可选 gemini-cli 的 A2A 协议（但其 SDK 0.16.x 尚不稳定，暂作演进储备）。

- **opencode 的 durable 输入模型（steer 合并 / queue FIFO + advisory wake）** 解决"用户在 agent 忙时插话"与崩溃恢复，是多端并发输入的正解——日常规划/长任务尤其需要可中断、可排队、可恢复的输入语义。
- **fork/分支学 codex 的 ForkSnapshot 或 pi 的 navigateTree**（非线性历史树对调研/规划探索不同路径有价值）。

## 拍板：构建策略 = **自研内核 + 借用模块（语言用 TypeScript/Bun + Effect）**

**结论：不 fork 单一基座，也不纯从零；而是以 opencode 的 server/client 分离架构为"主骨架蓝本"，自研一个 TS 内核，按上表择优移植各仓的子系统设计。**

**强论证：**

1. **网页优先这一最高约束直接排除大半候选**。你的第一性需求是"网页端先发"。claudecode（网页靠 bridge 远程驱动 CLI，是妥协）、aider/MetaGPT（无 Web）、gemini-cli/codex（CLI 表层为主，网页要重写 UI 层）都不满足"网页原生优先"。**只有 opencode 和 OpenHands 是真正 server/client 分离、网页一等公民**——而 opencode 的架构最干净（一套类型化 server + SDK 生成 + SSE，TUI/Web/Desktop/CLI 零差异接入），是网页优先的标准答案。

2. **为什么不直接 fork opencode？** 三个硬伤：(a) 它正处 V1→V2 大重写中间态（task 子代理/LSP/plan_exit/share 发布多项 TODO 待移植），fork 即踩半成品；(b) 它深度绑定 Effect 4.x **beta** + Bun + 大量 catalog/patch，生态小众、升级风险高、团队上手慢；(c) 它**专精编码**，五大能力中写作/computer-use/日常规划几乎为零。fork 它等于继承一个半成品重写 + 仍要自建大半能力层。

3. **为什么不 fork codex？** 它架构最干净、沙箱最强，但 (a) Rust 内核迭代曲线陡、网页优先需快速迭代不匹配；(b) ~120 crate + Bazel/Cargo + 大量 OpenAI 内部基建，全量 fork 负担过大；(c) 主路径深绑 OpenAI Responses API，多 provider 要重抽象。**它的价值是"架构原则的蓝本"（内核即库 + 事件流协议 + 三平台沙箱 + multicall 打包），而非 fork 底座。**

4. **语言选 TypeScript/Bun**：网页优先意味着前后端同构（React/Solid 前端 + TS 后端共享类型）、生态最大、迭代最快；opencode/cline/pi/gemini-cli/qwen-code 五个 TS 仓提供海量可直接移植的 TS 实现（工具系统/压缩/provider/扩展），降低自研成本。**不选 Rust（codex，迭代慢）、不选 Python（MetaGPT/OpenHands/opensquilla，前后端割裂、网页要另写 TS 前端）**——但 opensquilla/OpenHands 的子系统设计（写作流水线/沙箱/会话/记忆）作为"设计参考"移植到 TS。

5. **自研内核的最小骨架（按上表组装）**：
   - 架构 = opencode 的 server/client 分离 + 类型化 SDK 生成（网页优先地基）；
   - 主循环 = claudecode async-generator 事件流（映射 SSE）+ pi 纯函数分层；
   - provider = opencode 独立 LLM 网关包 + pi ThinkingLevel；
   - 工具 = codex 四态可见性 + claudecode 元数据 + opensquilla 落盘投影；
   - 沙箱 = codex 三平台原生（这部分可考虑 Rust helper 二进制 + TS 主进程，沙箱必须强隔离，不妥协）；
   - 扩展 = claudecode 四层（Skills 是五大能力载体）+ codex MCP 双向；
   - 持久化 = opensquilla 乐观锁（多端并发刚需）+ opencode durable 输入；
   - 能力层 = opensquilla 写作/调研流水线 + OpenHands web computer-use + 自建日常规划（共同空白）。

**一句话**：网页优先把架构蓝本钉死在 opencode，但其半成品 + 编码专精 + beta 依赖三个硬伤决定了"借蓝本不 fork 底座"；用 TS/Bun 自研内核，把上表 12 个 best_pick 作为模块设计来源逐一移植，沙箱借 codex 思路（必要时 Rust helper），五大能力中写作/调研借 opensquilla、computer-use 借 OpenHands+cline、日常规划自建。
